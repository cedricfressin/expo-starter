# i18n Rules

## Stack

- **Lingui** (`@lingui/core` + `@lingui/react` + `@lingui/macro`) for all translations
- **`expo-localization`** for device locale detection (`getLocales()`, `getCalendars()`)
- Catalog format: `.po` files in `locales/` — commit to git

## Usage

- `<Trans>` macro for JSX text — `t` macro for attributes and non-JSX strings
- `plural()` / `select()` macros for pluralization and gender — NEVER branch manually
- Parameterized messages: `` t`Hello ${name}` `` — NEVER concatenate translated strings
- Import `i18n` from `~/lib/i18n` — single source of truth for locale state

## Formatting

- `Intl.DateTimeFormat` / `Intl.NumberFormat` with `'default'` locale — NEVER hardcode date/number formats
- Currency, dates, numbers: always locale-aware via `Intl` APIs

## RTL

- Use logical properties: `ps-`/`pe-`, `ms-`/`me-`, `start`/`end` — NEVER `left`/`right` for layout
- Enable RTL in `app.json` → `expo.extra.supportsRTL: true`

## Catalog Management

- `bun x lingui extract` to sync source strings → `.po` files
- `bun x lingui compile` to generate optimized runtime catalogs
- Add new languages: import `.po` in `lib/i18n.ts` and add to `catalogs` record

## NEVER

- **NEVER** hardcode user-facing strings — use `<Trans>` or `t` macro
- **NEVER** concatenate translated strings — use parameterized messages
- **NEVER** use `i18n._()` directly — use `t` macro for extraction support
- **NEVER** hardcode date/number formats — use `Intl` APIs
- **NEVER** use `left`/`right` in layout — use logical properties (`start`/`end`)
